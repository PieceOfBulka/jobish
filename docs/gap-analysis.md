# Gap-анализ — реализовано vs документ

Сверка фактической кодовой базы (вертикальный срез MVP) с требованиями исходного
документа (см. `functionality.md`, `requirements-v2.md`, `architecture.md`).
Дата сверки: 2026-06-20.

## Контекст: два уровня «дойти до документа»
Документ описывает **распределённую enterprise-систему** (FastAPI + Kong +
микросервисы + PostgreSQL + Redis + S3 + парсер hh.ru; 5 чел / 11 мес).
Реализован осознанно упрощённый **монолит-срез** (Next.js + Prisma + SQLite,
моки вместо нереализуемых интеграций) — см. `audit.md §1`. Поэтому пробелы делятся
на (а) **функциональные** (реализуемо в текущем стеке) и (б) **архитектурные /
прод-интеграции** (крупный объём, по согласованию).

## План приведения к документу (фазы)

### Этап 1 — функциональные пробелы MVP (реализуемо в текущем стеке)
1. **ФТ-2.1/2.3 тестовая система**: банк профтеста 30–60 вопросов; типы вопросов
   (multiple/scale/text) + веса; сохранение прогресса и история попыток.
   Затрагивает `schema.prisma` (Test/Question с `question_type`, `weight`),
   `lib/orientation.ts`, `OrientationQuiz.tsx`, `seed.ts`.
2. **ФТ-1.1 SSO-задел**: `Auth_Provider` в схему + мок-кнопки google/vk/telegram
   с явной пометкой «демо».
3. **ФТ-3.1 сводка рынка**: медиана/перцентили + разрез по городу в
   `MarketAnalytics` и на экране аналитики.
4. **US17 минимальная админка**: раздел `/(admin)` — модерация пользователей
   (блокировка, верификация, назначение трека) + наполнение банка вопросов.

### Этап 2 — нефункциональные / UX
5. **NFR_U3 i18n RU/EN**: next-intl или лёгкий словарь, вынос строк.
6. Логирование, rate-limit на уровне роутов (замена Kong для среза).

### Этап 3 — приближение к прод-архитектуре (по согласованию, крупный объём)
7. SQLite → PostgreSQL (изменения Prisma минимальны).
8. Реальный hh.ru API (официальный, не скрапинг).
9. РФ-LLM (GigaChat/YandexGPT) вместо OpenRouter — под 152-ФЗ.
10. Реальные платежи, S3, email.
11. Кабинет коуча (US23–31) — отдельный крупный блок (Release X).

## Рекомендация по архитектуре
Для учебного среза — **оставаться на монолите** (Next.js+Prisma), закрывая
функциональные пробелы; PostgreSQL и прочее из Этапа 3 — только если цель ставит
прод-готовность. Это сохраняет работающее приложение и не распыляет объём.
