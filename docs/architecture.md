# Архитектура — Jobish

## Архитектура, заявленная в документе (L2)
SPA-фронт (React+TS) + Admin Panel (React+TS) → **Kong API Gateway** (SSL termination, auth, rate-limit) → **Jobish Backend API** (Python/FastAPI, монолит, ФТ 1–7) ↔ **AI Orchestrator** (Python/LangChain, выделен из-за долгих SSE-стримов LLM) ↔ **Data Parser** (Python/Scrapy+Selenium, парсинг hh.ru/курсов) → **PostgreSQL** (единая БД, схемы по доменам) + **Redis** (кеш справочников + JWT refresh/blacklist + очередь) + **S3** (вложения чата, резюме PDF/DOCX ≤10 МБ) + **LLM** GPT-4o-mini.

### Обоснования из документа (важные)
- FastAPI асинхронный — для SSE-стрима ответов ИИ и параллельных вызовов внешних сервисов.
- AI Orchestrator вынесен в отдельный контейнер: LLM-запрос 2–10 c, долгие SSE заняли бы thread pool (NFR_P1).
- Data Parser изолирован: внешние источники нестабильны, чтобы не ронять основной API (NFR_R2). На MVP — инструмент для LLM получить актуальные данные, без отдельного ETL.
- Единая PostgreSQL: JOIN без сетевых вызовов; каскадное удаление под GDPR/152-ФЗ; партиционирование диалогов по дате.
- Redis: кеш Profession/Profession_Skill (редко меняются, часто читаются); JWT в Redis без обращения к БД.

## НФТ (нефункциональные требования)
- **Надёжность:** доступность 99.5%, изоляция нестабильных компонентов.
- **Безопасность:** HTTPS/TLS ≥1.2, аутентификация на Gateway, GDPR + 152-ФЗ (удаление ПДн по запросу).
- **Производительность:** стабильно при 500 одновременных пользователях, масштабирование до 10 000 MAU без смены архитектуры.
- **Удобство:** адаптивность под устройства (NFR_U2), мультиязычность RU/EN (NFR_U3).
- **Поддерживаемость:** обновления без остановки сервиса (NFR_SP1).

## Реализуемая архитектура (предложение для веб-сайта-среза) — НА ПОДТВЕРЖДЕНИИ
Вместо распределённой системы — **монолит** с честными слоями, чтобы получить реально работающий сайт:
- **Next.js (React + TypeScript) + Tailwind** — UI + серверные API-роуты (вместо отдельных React-фронта и FastAPI).
- **Prisma + SQLite/Postgres** — данные и модель.
- **Своя JWT-аутентификация** — регистрация/логин/профиль.
- **AI-чат:** реальный LLM при наличии ключа ИЛИ детерминированный мок-движок (дефолт — мок, работает без ключей/интернета).
- **Тесты:** Vitest (unit/интеграционные) + Playwright (E2E).

### Что мокаем (с пометкой в UI)
Платежи/эквайринг, live-данные рынка труда (сид вместо скрапинга hh.ru), email/Zoom-уведомления, S3 (локальное хранилище), прод-SSO.

### Вне объёма среза (только как задел)
Kong, отдельные микросервисы, реальный Scrapy/Selenium, кабинет коуча и полная админка (Release X).

## Логическая модель данных (ключевые сущности из документа)
User, Profile, Profession, Profession_Skill (грейды), CareerGoal, Roadmap/RoadmapStep, Skill/SkillProgress, Test/Question/Attempt (+freeze), LearningMaterial, ChatSession/ChatMessage, Subscription/Plan/Payment, MarketAnalytics. Сертификаты решено НЕ выделять в отдельную сущность (временный характер данных).

## Архитектурные риски (см. также docs/audit.md)
1. **152-ФЗ ↔ OpenAI/S3** — передача ПДн в США нарушает локализацию; для прода нужен GigaChat/YandexGPT + сервер РФ.
2. **Скрапинг hh.ru** запрещён ToS → официальный API / моки.
3. **99.5% при одной PostgreSQL** — SPOF, нужна HA-реплика.
4. **Себестоимость LLM 0.029 ₽/запрос** занижена (не учтён контекст диалога/roadmap).
5. **ML-ресурс** (2.9 чел.-мес.) мал для ядра-AI.
